# 个人实训总结报告

## 一、项目概况

本次实训我独立完成了「TaskPlanner 任务看板系统」的开发。项目基于 Next.js 16 (App Router) + TypeScript + TailwindCSS + Mongoose + MongoDB 技术栈，实现了三栏看板（待办/进行中/已完成）的核心功能，并附加了仪表盘日历、深色模式切换、手绘鱼缸互动等特色功能。

项目代码托管在 GitHub，部署至 Railway 平台，共包含 3 个前端路由页面、11 个后端 API 接口、16 次 Git 提交记录，代码结构符合 Next.js 规范。

## 二、遇到的最大困难及解决过程

### 2.1 深色模式的 hydration 错误

在实现深色模式切换功能时，遇到了严重的 Hydration 错误。原因是 Next.js 服务端渲染时无法访问 localStorage，导致服务端渲染亮色模式，而客户端首次渲染时通过 useState 惰性初始化读取 localStorage 后直接使用深色模式，两边渲染结果不一致，React 报错并无法自动修复。

解决过程经历了多次迭代：一开始尝试在 ThemeContext 中使用 mounted 状态延迟渲染，但问题仍然存在，因为 `isDark` 的惰性初始化在客户端第一次渲染时就已经返回了深色值。最终采用了两层防御策略——全局布局的 `<html>` 标签添加 `suppressHydrationWarning` 属性，同时在每个使用主题色的组件中增加 `mounted` 判断，在 hydration 完成前使用亮色模式的 fallback 值。这个方案虽然牺牲了首次加载时的轻微闪烁，但彻底解决了 hydration 不匹配的问题。

### 2.2 部署后的域名访问问题

项目在 Railway 上部署成功后，日志显示应用运行正常（`Ready in 76ms`），Raiway 也生成了域名并验证了自定义域名的 DNS 记录。但无论是 Railway 的官方二级域名 (`up.railway.app`) 还是自定义的域名 (`www.todoboard.top`)，在国内网络环境下都无法访问。

排查过程从检查服务日志（发现应用正常运行）、验证域名 DNS 记录（通过 Railway 的 TXT 验证）、到使用第三方 DNS 检测工具确认域名已全球生效。最终确认是 Railway 的服务器节点位于美国，国内网络对部分境外 IP 的访问受限。这个限制超出了代码层面的解决范围，最终决定以本地录屏作为演示方式的替代方案。

### 2.3 摸鱼鱼缸的性能与交互优化

手绘鱼缸功能经历了三次重构：从最初简单的 6 条鱼自动游动，到用户绘制 + 投放鱼缸，再到所画即所得的完整交互流程。最大挑战在于如何让鱼的游动看起来自然真实，同时将用户绘制的图案作为鱼的身体纹理应用到 Canvas 动画中。最终通过贝塞尔曲线绘制鱼尾、缓动转向算法、以及周期性随机游走逻辑，实现了较为自然的游动效果。

## 三、对 AI 辅助编程的心得体会

### 3.1 AI 作为编程伙伴而非替代者

在本次实训中，AI（Codex）承担了代码生成、问题排查、方案设计等角色。但 AI 并非简单的代码生成器——它更像一个有经验的编程伙伴。AI 生成的代码需要人工理解和验证，尤其是在处理框架特有的机制（如 Next.js 的 Server Action、App Router 路由约定）时，AI 生成的代码可能不完全符合作业规范或实际需求。

### 3.2 提问质量决定回答质量

使用 AI 辅助编程的实践中，我发现把一个问题拆解为多个小问题并逐步追问，比一次性提一个笼统的问题效果更好。例如在实现深色模式时，我没有一次性问"怎么实现深色模式"，而是分步提问——"怎么创建主题上下文"、"组件如何适配深色"、"hydration 错误怎么修"。每一步的 AI 输出都更精准，也更易于理解和整合。

### 3.3 AI 的局限性

AI 在以下方面仍有明显局限：无法处理网络环境相关的部署问题（如 Railway 域名在国内无法访问）、对特定版本框架的 API 变化不敏感、生成的代码有时缺少异常处理。这些都需要开发者的人工判断和补全，也说明 AI 辅助编程的本质是"人主导、AI辅助"。

## 四、个人贡献

- **项目架构设计**：确定了 Next.js App Router + MongoDB 的技术方案，设计了 3 个前端路由和 11 个 API 端点的接口规范
- **核心功能实现**：完成了任务 CRUD、三栏看板、排序筛选、逾期标红、深色模式、日历视图、一键清理等全部功能
- **摸鱼鱼缸系统**：独立设计并实现了 Canvas 鱼缸动画，包含手绘鱼、命名投放、喂食长大、数据持久化等完整交互
- **工程化实践**：使用环境变量管理配置、try-catch 覆盖全部 API 异常处理、Git 提交规范（feat/fix/docs 前缀）、CSS 变量管理主题色
- **技术债务清理**：修复了 7 个 Code Review 中发现的代码问题，包括 hydration 错误、状态更新不同步等
- **文档建设**：编写了 README.md、API.md、prompt_log.md 三份完整文档

## 五、总结

本次实训让我完整经历了一个 Web 全栈项目的开发全流程——需求分析、技术选型、编码实现、测试验证、部署上线。过程中遇到了不少技术挑战，从 Next.js 的 hydration 机制到 MongoDB 的聚合查询，再到 Railway 的部署配置，每一个问题的解决都加深了我对技术栈的理解。

更重要的是，通过深度使用 AI 辅助编程工具，我体会到了"人机协作"的开发模式如何提升效率，也认识到 AI 并非万能——代码审查、架构设计、问题排查仍需开发者的经验和判断。这些经验对我未来的工程实践具有重要的参考价值。
